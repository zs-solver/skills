# 需求澄清

## 输入方式

两种主要输入，选择对应流程：

### A. 用户直接描述需求
用户在对话中描述需求（可能是一句话，也可能是详细描述）。
→ 进入第 1 步。

### B. 指定 inbox 文件
用户指定 `specs/inbox/` 下的速记文件。
→ 读取文件内容作为需求描述，进入第 1 步。

## 工作流程

### 1. 确认理解
复述用户的需求，确认理解是否正确。

### 2. 结构化提问
根据需求的模糊程度，选择性提问（不必每个都问）：
- 用户是谁？使用场景是什么？
- 输入是什么？输出是什么？
- 有哪些边界情况？
- 跟现有功能有什么关联？
- 有没有性能要求？
- 优先级是什么？（高/中/低）

### 3. 评估复杂度
需求澄清完成后，评估任务复杂度：
- 改动 ≤ 2 个文件、逻辑清晰无歧义 → `simple`（可直接实现）
- 否则 → `needs-design`（需要设计方案）

告知用户评估结论。

### 4. 输出需求文档

**创建文件夹和文件：**
`specs/tasks/yyyyMMdd_HHmm-任务名/requirement.md`

**如果输入来自 inbox 文件：** 将原 inbox 文件的 frontmatter status 更新为 `promoted`，并添加 `promoted_to` 字段指向新文件夹路径。

**Frontmatter 格式：**
```yaml
---
title: 功能名称
created: 2026-03-10
status: requirement
priority: 高
complexity: simple
category: （AI 根据内容判断，收尾时与用户确认）
tags:
  - 标签1
  - 标签2
---
```

**文档结构：**
```markdown
# [功能名称] 需求

## 核心需求
[1-3 句话说清楚要做什么]

## 使用场景
[谁在什么情况下用]

## 输入输出
- 输入：...
- 输出：...

## 边界条件
- [条件1]
- [条件2]

## 相关文档与资源
- [[../其他任务/requirement.md]] — 本任务修改的功能在该需求中首次定义
- [[docs/xxx.md]] — 某规则的权威来源
- 现有指标计算 API（`/api/indicators`）— 底层数据和计算逻辑可复用
- 参数转换模块（`backend/agent/.../parameter_converter.py`）— 名称到代码的转换
- [[../后续任务/requirement.md]] — 该任务消费本任务的输出格式，修改本任务时需检查其是否受影响
```

> **相关文档与资源格式**：每行写一个引用 + 一句话说明关系。有独立文档的用 wikilink，代码路径和接口用反引号标注。后续方案设计/实现阶段根据关系描述判断是否需要读取，避免全部加载浪费上下文。

### 5. 收尾整理

需求文档创建完成后，依次检查以下各项，合并为一次提问：

#### 5a. 确认分类

AI 在写文档时已根据内容预填了 `category`，让用户确认或修改。

#### 5b. 整理散落的引用文件

检查用户在本次对话中引用的外部文件（通过 `@文件路径` 或直接提及的文件）。

**判断条件**：引用的文件不在 requirement.md 所在的文件夹中。

**如果发现散落文件**：
- 询问用户是否将这些文件移动到 requirement.md 所在的文件夹中
- 推荐：**移动**（集中管理，便于后续方案设计和回溯）
- 如果根据实际情况有更合适的位置（如文件属于另一个已有任务），也可推荐替代方案
- 用户同意后执行移动，并在 requirement.md 的 `## 相关文档与资源` 中添加条目和关系说明

#### 5c. 双向引用已有需求

如果新需求引用了已有任务的文档（如 `specs/tasks/xxx/requirement.md`），检查被引用的文档是否也反向引用了新需求。

**如果缺少反向引用**：
- 询问用户是否在被引用文档的 `## 相关文档与资源` 中添加对新需求的引用（wikilink + 关系说明）
- 推荐：**添加**（保持双向可追溯，便于查看关联任务）

#### 5d. Inbox 文件清理（仅 inbox 模式）

如果输入来自 inbox 文件，当前流程已将其标记为 `promoted`。进一步询问如何处理，提供四个选项：

| 选项 | 说明 |
|------|------|
| **整理并删除（推荐）** | 将 inbox 原始内容作为附录整理到 requirement.md，然后删除 inbox 文件 |
| 仅删除 | 直接删除 inbox 文件，不整理原始内容（requirement.md 已独立完整时适用） |
| 仅整理 | 将原始内容整理到 requirement.md，但保留 inbox 文件（promoted 状态） |
| 不处理 | 保持现状（inbox 文件留在原处，维持 promoted 标记） |

**附录写法要求**：
- **不要**写 `来源：specs/inbox/xxx.md` ——选择"整理并删除"后文件已不存在，引用无意义
- 附录标题统一用 `## 附录：原始需求速记`
- 只保留原始内容的核心信息（改动要点、对比示例等），不需要完整复制

**用户后续补充**：如果用户在需求文档创建后又补充了说明或修正（如"所有维度都要有学校id和年份"），应将这些补充**原文追加**到附录末尾，保留用户的原始表述。

#### 合并提问示例

```
收尾整理：

1. 发现以下引用文件不在任务文件夹中：
   - specs/xxx.md
   建议移动到任务文件夹中集中管理。是否移动？（推荐：是）

2. 新需求引用了 specs/tasks/xxx/requirement.md，
   建议在该文档中添加对本需求的反向引用。是否添加？（推荐：是）

3. inbox 文件 specs/inbox/xxx.md 如何处理？
   a. 整理到 requirement.md 并删除（推荐）
   b. 仅删除
   c. 仅整理到 requirement.md，保留 inbox
   d. 不处理
```

> 注意：以上三项按实际情况出现，没有的项跳过，不要问不存在的问题。

## 重要原则

- 只关注"要什么"，不关注"怎么做"
- 不包含任何技术实现细节（如文件路径、函数名、实现步骤）
- 不包含方案选择或技术决策
- 技术细节留给后续的方案设计阶段
- **外部路径必须保留绝对路径**：当需求涉及本项目之外的文件或目录时（如其他仓库、外部数据源），在需求文档中**必须使用完整的绝对路径**，严禁简化为相对路径或缩写。原因：后续会话无法从缩写推断出实际位置，导致找不到文件。本项目内部的路径可以用相对路径。

## 过程中的反思沉淀

在需求澄清过程中，如果用户纠正了 AI 的理解偏差、反复强调某个约定、或发生了需求变更，应触发 `retrospective` skill 记录到任务文件夹的 `retrospective.md`。发现即记录，不必等到需求澄清结束。

## 版本变更规则

- 初始需求文件为 `requirement.md`（无版本号，视为 v1）
- 如果需求发生重大变更，新建 `requirement-v2.md`，不修改原文件
- 当前生效版本：版本号最大的文件；没有版本号则直接使用该文件
