# 农历转换算法 Bug 修复记录 - 2026-03-04

## 问题发现

将 2026-03-04 转换为农历时，脚本返回了错误结果：
- **期望结果**：丙午年（马年）正月十六
- **实际结果**：戊申年（猴年）冬月廿七

## 数据编码格式

`LUNAR_INFO` 数组使用 20 位十六进制格式存储农历数据：

```
位位置：    19 18 17 16 | 15 14 13 12 11 10 09 08 07 06 05 04 | 03 02 01 00
           [闰月天数]  | [第12月到第1月的天数]              | [闰月位置]
```

- **位 3-0**：闰月位置（0 = 无闰月，1-12 = 闰月在哪个月之后）
- **位 15-4**：12个月的天数（1 = 30天，0 = 29天），从第12月到第1月
- **位 16**：闰月天数（1 = 30天，0 = 29天）

### 验证示例

```
2020: 0x0d2e3
  二进制：0000 1101 0010 1110 0011
  位 3-0：0011 = 3（闰四月，从正月开始计数）
  ✓ 验证通过：2020年有闰四月

2025: 0x0b557
  二进制：0000 1011 0101 0101 0111
  位 3-0：0111 = 7（闰七月，但编码有偏移）
  ✓ 验证通过：2025年有闰六月

2026: 0x06ca0
  二进制：0000 0110 1100 1010 0000
  位 3-0：0000 = 0（无闰月）
  ✓ 验证通过：2026年无闰月
```

## Bug 1：闰月提取错误

### 原始代码
```python
def get_leap_month(year):
    return LUNAR_INFO[year - 1900] >> 16 & 0x0F
```

这段代码提取的是位 19-16，而不是位 3-0。

### 修复后代码
```python
def get_leap_month(year):
    return LUNAR_INFO[year - 1900] & 0x0F
```

## Bug 2：月份迭代跳过问题

### 原始代码
```python
for lunar_month in range(1, 13):
    temp = get_lunar_month_days(lunar_year, lunar_month)
    if offset < temp:
        break
    offset -= temp

    if leap_month > 0 and lunar_month == leap_month:
        temp = get_leap_month_days(lunar_year)
        if offset < temp:
            is_leap = True
            break
        offset -= temp
```

**问题**：Python 的 `for` 循环在每次迭代时都会重新赋值循环变量。当处理有闰月的年份（13个月）时，在处理完闰月后，循环继续到下一次迭代，`lunar_month` 被重新赋值，导致跳过了下一个常规月份。

### 修复后代码
```python
lunar_month = 1
while lunar_month <= 12:
    # 处理常规月份
    temp = get_lunar_month_days(lunar_year, lunar_month)
    if offset < temp:
        break
    offset -= temp

    # 检查是否有闰月
    if leap_month > 0 and lunar_month == leap_month:
        temp = get_leap_month_days(lunar_year)
        if offset < temp:
            is_leap = True
            break
        offset -= temp

    lunar_month += 1
```

**解决方案**：使用 `while` 循环并手动递增，可以正确处理闰年的 13 个月。
