# 反思沉淀

## 工作流程

### 1. 回顾实现
- 读取任务文件夹下的 requirement.md 和 design.md（取版本号最大的）
- 查看实际代码改动
- 对比方案和实现的差异

### 2. 提取任务级信息
只记录以下内容：
- **方案变更**：实际实现与 design.md 的偏差及原因
- **踩过的坑**：现象 → 原因 → 解决方式

不记录：
- 代码做了什么（看代码就知道）
- 函数签名、参数列表（看代码就知道）
- 实现细节（看代码就知道）

### 3. 输出任务级复盘文档
写入同一任务文件夹下的 `retrospective.md`。

更新 requirement.md 的 frontmatter `status` 为 `done`，添加 `completed` 字段。

文档结构：
```markdown
# [功能名称] 复盘

## 方案变更
[实际实现与 design.md 的偏差及原因，无变更则写"无"]

## 踩过的坑
- [坑1]：现象 → 原因 → 解决方式
- [坑2]：现象 → 原因 → 解决方式
```

### 4. 分流可复用知识
检查本次实现中是否产生了跨任务可复用的知识，按类型分流：

**a) 技术陷阱 / 踩坑经验 → `specs/经验/`**
- 如果已有相关文件（如 `Bash工具执行Python陷阱v1.md`），追加内容
- 如果没有，新建文件，命名格式：`主题描述.md`

**b) 用户偏好 / 开发约定 → `CLAUDE.md` 或 `specs/通用规则.md`**
- 如"用户倾向于把设计文档讨论得很详细"→ 写入通用规则
- 如新的编码约定 → 写入 CLAUDE.md

**c) 系统架构知识 → `docs/` 下对应文档**
- 如"学院维度用 dwh 字段"→ 更新到 docs/indicator/ 相关文档

**d) 不属于以上分类 → 询问用户放在哪里**

每条分流都告知用户：更新了哪个文件、添加了什么内容。

## 重要原则

- 精简至上，只记录决策和踩坑
- 任务级复盘留在任务文件夹，通用知识分流到对应位置
- 分流前询问用户确认，避免误改重要文档
