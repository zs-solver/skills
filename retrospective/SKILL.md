# 反思沉淀

## 工作流程

### 1. 确认上下文
- 过程中触发（当前会话已有上下文）→ 直接进入第 2 步
- 完成后触发或新会话 → 读取任务文件夹下的 requirement.md 和 design.md（取版本号最大的），查看代码改动

### 2. 提取值得记录的信息
只记录以下类型：
- **用户纠正 / 强调事项**：用户纠正了 AI 的理解偏差，或反复强调的约定
- **决策变更**：需求或方案发生变更及原因
- **踩过的坑**：现象 → 原因 → 解决方式

不记录：
- 代码做了什么（看代码就知道）
- 函数签名、参数列表（看代码就知道）
- 实现细节（看代码就知道）

### 3. 写入 retrospective.md
写入同一任务文件夹下的 `retrospective.md`。
- 如果文件已存在 → 追加新内容
- 如果文件不存在 → 新建

每条记录格式：
```markdown
## [YYYY-MM-DD 阶段标识]
- **[类型]**：[具体内容]
```

> 阶段标识：需求澄清 / 方案设计 / 实现 / 复盘
> 类型：用户纠正 / 决策变更 / 踩坑 / 强调事项

### 4. 收尾（仅任务完成时）
更新 requirement.md 的 frontmatter `status` 为 `done`，添加 `completed` 字段。

### 5. 分流可复用知识（可选）
过程中的小记录通常无需分流。仅在信息具有跨任务复用价值时，按类型分流：

**a) 技术陷阱 / 踩坑经验 → `specs/经验/`**
- 如果已有相关文件（如 `Bash工具执行Python陷阱v1.md`），追加内容
- 如果没有，新建文件，命名格式：`主题描述.md`

**b) 用户偏好 / 开发约定 → 写入 `CLAUDE.md`**

**c) 系统架构知识 → `docs/` 下对应文档**
- 如"学院维度用 dwh 字段"→ 更新到 docs/indicator/ 相关文档

**d) 不属于以上分类 → 询问用户放在哪里**

每条分流都告知用户：更新了哪个文件、添加了什么内容。

## 重要原则

- 精简至上，只记录后续 AI 无法从代码推断的信息
- 过程中触发时保持轻量，不打断主流程
- 任务级记录留在任务文件夹，通用知识按需分流
- 分流前询问用户确认，避免误改重要文档
