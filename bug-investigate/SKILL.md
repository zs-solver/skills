# Bug 排查

## 工作流程

### 1. 确认现象
复述用户描述的问题，确认：
- 具体现象是什么（看到了什么、预期是什么）
- 复现条件（什么操作下出现、是否稳定复现）
- 用户已经排除的方向（如果有）

### 2. 排查定位
先读代码，再分析——永远不要在没看代码的情况下给出诊断。

排查过程中：
- 每发现一个关键线索，向用户确认是否符合实际行为
- 用户补充的额外信息是重要线索，后续写入文档时原文保留
- 多个异常点时先理清因果关系，区分"根本原因"和"下游受影响的表现"

需要查看服务器真实数据或调用接口验证时，使用 `server-inspect` skill。

### 3. 汇报根因
向用户说明：
- 根因是什么（推测须标注"⚠️ AI 推测，未经验证"）
- 影响范围
- 排查中顺带发现的关联问题

### 4. 评估修复路径

| 情况 | 建议路径 |
|------|---------|
| 改动 ≤ 2 个文件、逻辑清晰 | 直接修复 → `/implementation-verify` |
| 改动多文件、有设计决策 | 先输出调查报告 → `/solution-design` |
| 根因不明、需要更多信息 | 继续排查，或请用户补充复现细节 |

### 5. 输出调查报告
写入任务文件夹下的 `investigation.md`。模板见 `specs/templates/调查报告模板.md`（如不存在则按以下结构写）：

问题现象 → 排查过程（每步：看了什么 → 发现了什么）→ 根因 → 影响范围 → 关联发现 → 用户提供的关键信息（原文）→ 修复建议

### 6. 创建 requirement.md（如尚未创建）
如果任务文件夹下还没有 `requirement.md`，在输出调查报告的同时创建一个简短的 `requirement.md`，包含 frontmatter 和问题描述，通过 wikilink 引用 `investigation.md`。

### 7. 后续衔接
- 简单修复 → 直接实现 → `/implementation-verify`
- 复杂修复 → `/solution-design` → 实现 → `/implementation-verify`
- 修复完成 → `/retrospective`

## 重要原则

- **保留用户原话**：用户描述现象、补充线索时的原始表述必须保留，不要改写
- **不急于修复**：定位根因后先汇报，等用户确认再决定是直接修还是走方案设计
- **区分因果**：多个异常点时先理清上下游关系，不要平行列举
- 排查中用户纠正分析方向或否决假设时，触发 `retrospective` skill 记录
