# 方案设计

## 工作流程

### 1. 理解需求
读取任务文件夹下的 requirement.md（取版本号最大的），确认理解是否正确。

### 2. 查看文档
- 读取 requirement.md 的 `## 相关文档与资源`，根据关系描述判断哪些需要读取（不必全读）
- 读取 `docs/` 中相关文档

### 3. 查看代码
- 优先使用 Explore agent 搜索相关代码
- 顺着调用链读代码
- 读完整实现，不只看函数签名

### 4. 验证一致性
检查文档和代码是否一致，不一致就反馈。

### 5. 提出方案选项（如果有多个）
如果有多个可行方案，先向用户说明：
- 方案1：[简述]，优点：...，缺点：...
- 方案2：[简述]，优点：...，缺点：...
- 推荐：[哪个方案，为什么]

让用户选择后再继续。

### 6. 提出待确认问题
把代码中发现的关键决策点抛给用户：
- "我看到 `file.py:42` 的 `func()` 是同步的，需要改异步吗？"
- "这个数据在 `data/cache/`，新数据也放这里吗？"
- "要新建文件还是改现有文件？"

提问要具体到文件路径和行号。

### 7. 确认实现偏好
根据方案规模和类型，询问用户：
- "新增代码放新文件还是现有文件？"
- "这个函数预计 XX 行，要拆成独立文件吗？"
- "这是工具脚本，要加 if __name__ 测试吗？"
- "这些参数要提取到配置文件吗？"

### 8. 用户确认
等待用户确认方案、回答问题、确认实现偏好。

### 9. 封存上下文
文档写入前，询问用户：
- "对话中还有什么关键信息没有写进文档？"
- "后续实现时是否需要注意什么额外约定？"

将回答写入设计文档的「补充说明」章节。

### 10. 输出最终设计文档
写入同一任务文件夹下的 `design.md`。

更新 requirement.md 的 frontmatter `status` 为 `designed`。

文档结构：
```markdown
# [功能名称] 技术方案

## 方案概述
[1-3 句话说明最终方案]

## 核心改动
每个改动点必须自包含，让实现会话不需要预先探索代码库就能直接动手：
- 文件：`path/to/file.py`
- 改动位置：[函数名或行号范围]
- 当前代码：[贴出需要修改的关键代码片段，标注行号]
- 改为：[目标代码或明确的改动描述]

## 实现步骤
1. [具体到文件和函数]
2. [关键代码示意]

## 关键决策
- [决策点1及理由]
- [决策点2及理由]

## 实现约定
- [实现偏好结论]

## 补充说明
- [对话中额外的上下文信息]
```

## 重要原则

- 设计文档只写最终确定的方案，不写方案对比
- 方案对比在第 5 步口头讨论，不写入文档
- 设计文档应该是可以直接实现的最终版本，实现会话读完文档就能动手，不需要再探索代码库
- 文档定稿后建议新开会话进行实现，所有必要信息已在文档中

## 过程中的反思沉淀

在方案设计过程中，如果用户纠正了 AI 的理解、否决了某个方案、强调了设计约束、或发现了技术陷阱，应触发 `retrospective` skill 记录到任务文件夹的 `retrospective.md`。发现即记录，不必等到方案定稿。

## 版本变更规则

- 初始方案文件为 `design.md`（无版本号，视为 v1）
- 如果方案需要推翻重来，新建 `design-v2.md`，不修改原文件
- 当前生效版本：版本号最大的文件；没有版本号则直接使用该文件
