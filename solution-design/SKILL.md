# 方案设计

## 工作流程

### 1. 理解需求
读取需求文档（如果有），确认理解是否正确。

### 2. 查看文档
- 读取 `CLAUDE.md` 或 `README.md`
- 读取 `specs/通用规则.md`
- 读取 `docs/` 中相关文档

### 3. 查看代码
- 搜索相关文件
- 顺着调用链读代码
- 读完整实现，不只看函数签名

### 4. 验证一致性
检查文档和代码是否一致，不一致就反馈。

### 5. 提出方案选项（如果有多个）
如果有多个可行方案，先向用户说明：
- 方案1：[简述]，优点：...，缺点：...
- 方案2：[简述]，优点：...，缺点：...
- 推荐：[哪个方案，为什么]

让用户选择后再继续。

### 6. 提出待确认问题
把代码中发现的关键决策点抛给用户：
- "我看到 `file.py:42` 的 `func()` 是同步的，需要改异步吗？"
- "这个数据在 `data/cache/`，新数据也放这里吗？"
- "要新建文件还是改现有文件？"

提问要具体到文件路径和行号。

### 7. 用户确认
等待用户确认方案和回答问题。

### 8. 输出最终设计文档
写入 `specs/XX-分类/yyyyMMdd_HHmmss-任务名/design.md`

文档结构：
```markdown
# [功能名称] 技术方案

## 方案概述
[1-3 句话说明最终方案]

## 核心改动
- 文件：`path/to/file.py`
- 改动位置：[函数名或行号范围]

## 实现步骤
1. [具体到文件和函数]
2. [关键代码示意]

## 关键决策
- [决策点1及理由]
- [决策点2及理由]
```

**重要原则：**
- 设计文档只写最终确定的方案，不写方案对比
- 方案对比在第5步口头讨论，不写入文档
- 设计文档应该是可以直接实现的最终版本
