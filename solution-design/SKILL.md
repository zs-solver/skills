# 方案设计

## 工作流程

### 1. 理解需求
读取需求文档（如果有），确认理解是否正确。

### 2. 查看文档
- 读取 `CLAUDE.md` 或 `README.md`
- 读取 `specs/通用规则.md`
- 读取 `docs/` 中相关文档

### 3. 查看代码
- 搜索相关文件
- 顺着调用链读代码
- 读完整实现，不只看函数签名

### 4. 验证一致性
检查文档和代码是否一致，不一致就反馈。

### 5. 提出疑问
把代码中发现的关键决策点抛给用户：
- "我看到 `file.py:42` 的 `func()` 是同步的，需要改异步吗？"
- "这个数据在 `data/cache/`，新数据也放这里吗？"
- "要新建文件还是改现有文件？"

提问要具体到文件路径和行号。

### 6. 说明方案
提出 2-3 个方案，对比优劣，推荐一个。

每个方案包含：
- 改哪些文件
- 改什么内容
- 关键代码示意
- 优缺点

### 7. 实现建议
询问用户实现层面的偏好：
- "这个函数会比较长，要拆成独立文件吗？"
- "这是工具脚本，要加 if __name__ 测试吗？"
- "这些参数要提取到配置文件吗？"

### 8. 输出文档
写入 `specs/XX-分类/yyyyMMdd_HHmmss-任务名/design.md`

文档结构：
```markdown
# [功能名称] 技术方案

## 需求概述
[1-3 句话]

## 相关代码
- `file:行号` - [作用]

## 方案对比
### 方案一
改动：...
优点：...
缺点：...

### 方案二
...

## 推荐方案
[选哪个，为什么]

## 实现步骤
1. [具体到文件和函数]

## 待确认
- [ ] [决策点]
```

只记录决策，不记录实现细节。
