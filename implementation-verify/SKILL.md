# 实现验证

## 核心目标

**证明代码修改生效了。**不是证明代码能跑，而是证明修改前后的行为发生了预期的变化。

AI 必须自行验证，不要让用户当测试员。

## 工作流程

### 1. 提取变更点

根据可用信息提取需要验证的变更点：

| 可用信息 | 提取方式 |
|---------|---------|
| 有 requirement.md + design.md | 从文档逐条提取"变更前 → 变更后" |
| 有 investigation.md（bug 修复） | 从根因和修复方案提取"bug 现象 → 修复后预期" |
| 只有代码改动 | 从 git diff 和对话上下文推导验证点 |

每个变更点必须形成一对：`[变更前] → [变更后]`。

### 2. 设计验证断言
每个变更点对应一个可自动验证的断言：

| 变更类型 | 断言示例 |
|---------|---------|
| 新增字段 | `assert "new_field" in response` |
| 删除字段 | `assert "old_field" not in response` |
| 字段类型变更 | `assert isinstance(response["field"], list)` |
| 新增行为 | 触发条件 → 检查新行为出现 |
| 过滤/筛选 | 遍历结果集，验证每条都满足过滤条件 |
| Bug 修复 | 复现原始操作 → 验证异常不再出现 |

关键：**断言必须能区分修改前和修改后**。如果一个断言在修改前也能通过，那它就不是有效的验证。

**条件分支变更**（降级、默认值、兜底逻辑等）必须构造两种输入分别测试：
- 正常路径：满足条件时的行为（如精确匹配 → 无 note）
- 分支路径：不满足条件时的行为（如降级匹配 → 有 note）

两条路径都必须验证，缺一不可。

### 3. 编写测试脚本
在任务文件夹下生成测试脚本，每个变更点一个测试函数。

脚本要求：
- 顶部 docstring 包含运行命令（假设用户在项目根目录）
- 如果依赖外部服务，先检查可用性，不可用时打印启动命令
- 每个测试清晰标注：**验证点名称 → 修改前是什么 → 修改后应该是什么 → 断言**
- Windows 上测试子进程不要用 `python -c` 内联，写成 `.py` 文件执行
- 输出格式：

```
----------------------------------------------------------------------
  验证: [变更点名称]
  修改前: [旧行为描述]
  修改后: [新行为描述]
----------------------------------------------------------------------

  结果: [实际输出的关键部分]
  判定: ✓ 修改已生效 / ✗ 修改未生效
```

### 4. 自主运行与循环修正
- **AI 自己运行测试脚本**，不要让用户手动执行
- 测试失败时，先怀疑测试脚本本身是否有问题（尤其是"所有测试都失败"的情况）
- 确认是代码问题后修正代码，重新运行测试
- 循环直到所有测试通过
- **测试通过后第一件事：确认项目代码的修复已到位**——测试是过程，项目代码的修复才是交付物

### 5. 前端变更的验证
纯前端变更无法用脚本验证时，给出具体的手动验证步骤：
```markdown
### 手动验证: [变更点名称]

**操作步骤**：
1. 打开 [页面 URL]
2. [具体操作]
3. [具体操作]

**修改前**：[旧的表现]
**修改后**：[应该看到什么]
**确认生效**：[看到什么就说明生效了]
```

### 6. 输出测试记录
- 测试脚本保留在任务文件夹下（不删除），每个脚本顶部说明：为什么写这个、发现了什么
- 运行结果写入任务文件夹下的 `测试记录.md`

### 7. 处理失败
发现问题只报告，不自动修复，等用户决定。

## 重要原则

- **AI 自主验证**：修完就测，不要改完代码就汇报让用户去试
- **每个变更点都必须有"修改前 vs 修改后"的对比**，这是验证的核心价值
- **断言必须在修改前会失败、修改后才通过**，否则不是有效验证
- **测试结果不符合预期时，先怀疑测试本身**：当所有方案都失败时，问题可能出在测试脚本上
- **交付意识**：测试是过程，项目代码的修复才是交付物。测试通过后立即确认生产代码状态
- 从需求文档推导测试内容，不从代码猜
- 测试脚本和结果留在任务文件夹

## 与其他 skill 的关系

```
功能实现：requirement-clarify → solution-design → 实现 → implementation-verify → retrospective
Bug 修复：bug-investigate → (solution-design) → 实现 → implementation-verify → retrospective
```
