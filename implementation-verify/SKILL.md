# 实现验证

## 核心目标

**证明代码修改生效了。**不是证明代码能跑，而是证明修改前后的行为发生了预期的变化。

## 工作流程

### 1. 提取变更点
读取任务文件夹下的 requirement.md 和 design.md（取版本号最大的），逐条提取：
- **变更前**是什么（旧的字段、旧的行为、旧的返回格式）
- **变更后**是什么（新的字段、新的行为、新的返回格式）

每个变更点必须形成一对：`[变更前] → [变更后]`。

### 2. 设计验证断言
每个变更点对应一个可自动验证的断言：

| 变更类型 | 断言示例 |
|---------|---------|
| 新增字段 | `assert "new_field" in response` |
| 删除字段 | `assert "old_field" not in response` |
| 字段类型变更 | `assert isinstance(response["field"], list)` |
| 新增行为 | 触发条件 → 检查新行为出现 |
| 过滤/筛选 | 遍历结果集，验证每条都满足过滤条件 |

关键：**断言必须能区分修改前和修改后**。如果一个断言在修改前也能通过，那它就不是有效的验证。

**条件分支变更**（降级、默认值、兜底逻辑等）必须构造两种输入分别测试：
- 正常路径：满足条件时的行为（如精确匹配 → 无 note）
- 分支路径：不满足条件时的行为（如降级匹配 → 有 note）

两条路径都必须验证，缺一不可。

### 3. 生成测试脚本
在任务文件夹下生成 `test_xxx.py`，每个变更点一个测试函数。

脚本要求：
- 顶部 docstring 包含运行命令（假设用户在项目根目录）
- 如果依赖外部服务，先检查可用性，不可用时打印启动命令
- 每个测试清晰标注：**验证点名称 → 修改前是什么 → 修改后应该是什么 → 断言**
- 输出格式：

```
----------------------------------------------------------------------
  验证: [变更点名称]
  修改前: [旧行为描述]
  修改后: [新行为描述]
----------------------------------------------------------------------

  结果: [实际输出的关键部分]
  判定: ✓ 修改已生效 / ✗ 修改未生效
```

### 4. 生成体验验证提问
除了自动化脚本，还要生成**可以直接复制到 Agent 中的自然语言提问**：

```markdown
### 验证 N：[变更点名称]

**提问**：
> [一句自然语言]

**修改前**：[Agent 以前会返回什么]
**修改后**：[Agent 现在应该返回什么]
**确认生效**：[看到什么就说明生效了]
```

这些提问应该写入提问手册或测试文档中。

### 5. 运行并报告
- **默认假设所有服务已在运行**，不要主动启停服务
- 只有当代码修改需要服务重启才能生效时，才提示用户重启
- 前端变更不需要脚本，直接告诉用户在页面上看什么
- 运行结果写入 `测试情况(日期-简述).md`

### 6. 处理失败
发现问题只报告，不自动修复，等用户决定。

## 重要原则

- **每个变更点都必须有"修改前 vs 修改后"的对比**，这是验证的核心价值
- **断言必须在修改前会失败、修改后才通过**，否则不是有效验证
- 从需求文档推导测试内容，不从代码猜
- 测试脚本和结果留在任务文件夹

## 与其他 skill 的关系

```
requirement-clarify → solution-design → 实现 → implementation-verify → retrospective
```
